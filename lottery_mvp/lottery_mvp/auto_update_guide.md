# 自动执行说明

当前项目的自动执行主方案是：

1. GitHub Actions 按固定时间自动运行 `python scripts/update_draws.py`
2. 更新完成后，把 `db/lottery.db` 自动提交回仓库
3. Streamlit 应用读取仓库中的最新 SQLite 数据
4. Actions 会自动生成一份中文更新汇总，方便你在运行记录里直接看结果

## 当前定时策略

- 执行时区按 `Asia/Shanghai`
- 每周一、二、三、四、六、日自动执行
- 每个执行日跑两次：
  - 21:45
  - 22:15

这样做的目的：

- 覆盖双色球和大乐透的开奖日
- 给官方页面延迟更新留一个补跑时间
- 避免非开奖日每天都产生无意义更新

## 相关文件

- 定时任务配置：`.github/workflows/update_lottery_data.yml`
- 更新脚本：`scripts/update_draws.py`
- 数据抓取与解析：`services/data_sources.py`
- 更新写库逻辑：`services/updater.py`

## 当前已接好的自动化行为

- 支持定时执行和手动触发 `Update Lottery Data`
- 每次执行后会生成中文汇总：
  - 新增写入数量
  - 已存在跳过数量
  - 更新失败数量
  - 双色球 / 大乐透逐项结果表
- 如果更新失败，还会自动补一段抓取诊断：
  - 官方页面能不能访问
  - 页面里有没有命中关键字段
  - 抓取器本身能不能成功解析
- 如果任一彩种抓取失败：
  - 数据库里成功写入的部分仍会保留并提交
  - 工作流最终会明确标红，便于你及时发现异常
- 如果两边都只是“已存在”，工作流会正常结束，不会重复提交数据库文件

## 使用方式

1. 把项目推到 GitHub 仓库
2. 确保仓库开启 GitHub Actions
3. 保持默认分支可被 Actions 写回
4. 等待定时执行，或在 Actions 页面手动触发 `Update Lottery Data`

## 自动执行后的效果

- 新开奖数据会写入 `db/lottery.db`
- 首页和分析页会显示最近更新状态
- 如果抓取失败，页面会显示失败说明
- GitHub Actions 的运行详情页会显示中文结果汇总
- 如果抓取失败，运行详情页还会附带“抓取源诊断”表

## 当前注意点

- 现在的数据抓取依赖官方页面结构，后面如果官方改版，需要同步调整解析逻辑
- 如果你后面改用 PostgreSQL，就不需要再把 SQLite 文件提交回仓库了，自动执行逻辑可以直接改成写远程数据库
